  What makes it one project, not three glued together

  Three things have to be shared, or you've just built a portfolio in a
  monorepo:
  1. One agent core — the same planner + tools serve the browser, phone, and
  meeting overlay.
  2. One conversation store — start on phone, finish in browser; the cockpit can
   replay/handoff.
  3. One observability + guardrails layer — every turn flows through the same
  trace + approval pipeline regardless of transport.

  Honest tradeoffs

  - Scope is real. This is 4–6 weeks of focused work, not a weekend. The phone
  bridge alone (SIP, μ-law, barge-in) is a project.
  - You'll feature-flag transports. WebRTC and WebSocket aren't combined —
  they're parallel surfaces. That's fine; the agent is what's unified.
  - The temptation is to build everything shallow. A stronger demo picks two
  surfaces deep (browser cockpit + phone bridge) and one mode (Realtime-2 with
  Translate as a toggle), all riding the shared spine. That still touches every
  guide.

  My recommendation

  Build the cockpit as browser (WebRTC) + phone (WebSocket) on a shared agent
  core, with the observability spine and approvals from day one. Add Translate
  as a mode switch in week 3. Skip the Zoom overlay until the core is solid — it
   adds platform integration pain without teaching you anything new about the
  API.
