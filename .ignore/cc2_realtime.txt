● Plain-English answers, grounded in your HVAC scenario:

  1. Voice-intent classifier                                                    
  In the HVAC demo, Reggie has to say something to approve a dangerous tool call
   — like "yes, do it" or "approve". Something has to listen to his words and
  decide:

  ▎ "Was that an approval, or just normal chatter?"

  That decider is the voice-intent classifier. Two ways to build it:

  - Local small model — a tiny speech recognizer + a list of trigger phrases
  runs on your own server. Fast (~50ms), free per use, but only handles phrases
  you've thought of in advance.
  - Round-trip to OpenAI — we send the audio back to OpenAI and ask "is this an
  approval?". Smarter (handles "yeah go for it" without you listing it), but
  slower (~500ms) and costs money per check.

  Default: local. Approvals happen many times per call; cheap+fast wins. The
  interface is clean, so we can swap to OpenAI later if local isn't good enough.

  2. S3-compatible blob store

  "S3" is Amazon's service for storing files (audio, images, PDFs) — separate
  from a database. "S3-compatible" means any storage service that speaks the
  same protocol: Amazon S3, Cloudflare R2, Backblaze B2, or self-hosted MinIO.

  The question is just: do we save the actual call audio to a file somewhere, so
   you can replay calls later?

  - Local disk — easiest, but fills up your server.
  - S3-compatible — scales, costs cents per GB, can be a separate provider.
  - Don't store audio at all — keep only the transcript. Cheapest, simplest, and
   avoids privacy/compliance headaches.

  Default: don't store audio in v1. Transcripts are enough for HVAC. Telehealth
  would later need real audio storage for compliance reasons, and we'd add S3
  then.

  3. Approval-on-voice phrase parsing

  This is how strict the voice-intent classifier should be. When Reggie says
  something to approve a truck dispatch:

  - Exact phrase match — only approve if he says exactly "Reggie, do it" (or
  whatever phrase you defined per tool). Anything else, even "yeah do it", gets
  ignored. Safe but rigid.
  - Fuzzy / intent match — accept "yes", "yeah", "go ahead", "do it",
  "approved", "ship it" — anything that means approval. Natural but riskier
  (someone says "yeah I had a long day" and the agent dispatches a truck).

  Default: exact phrase per tool, configurable. Each tool in approvals.yaml
  declares its own approval phrase — e.g. dispatch_truck requires "send the
  truck", schedule_move requires "move it". Verbose, but no false approvals.

  4. Frontend hosting

  The browser cockpit is a React app — it's just static HTML/JS/CSS files at the
   end of the day. Those files have to be served by some web server. Two
  options:

  - Served by the Node edge — the same Node process that handles audio also
  serves the frontend files. One fewer container.
  - Separate container — a tiny nginx (or Vite preview) container whose only job
   is serving the cockpit UI. Cleaner separation, easier to redeploy frontend
  without touching audio code.

  Default: separate container. The audio edge is latency-sensitive; we don't
  want frontend asset requests competing with WebRTC signaling. In
  docker-compose, this is one extra 8-line block — basically free.
