  Five concrete features that would bring gpt-realtime-whisper in — each additive, each with the exact files to touch:

  1. Voicemail / overflow handler — after-hours calls run on whisper instead of Aria. No tool risk, no hallucinated commitments, dispatcher reads transcript later.
  2. Bilingual transcript capture in translate mode — run whisper alongside translate so the source-language transcript is captured for audit.
  3. Note-taker mode — when the dispatcher takes a call directly (cockpit Talk), whisper silently transcribes both sides.
  4. Eval generation from real calls — make synthesize-eval CONV=<uuid> runs whisper over a past conversation and emits a YAML scenario.
  5. Compliance-grade audit transcripts — for regulated verticals (telehealth), whisper produces ground-truth transcripts that diff against the agent's view to flag divergences.

  The doc shows the shared "seam pattern" across all five: a new mode value → a model id mapped to that mode in the edge → a transcript-handling path that lands in app.turns → a cockpit affordance. Once any one
   of them ships, the next four become close to copy-paste.

 4. Eval generation from real calls — make synthesize-eval CONV=<uuid>

 whisper runs always-on; nightly diff flags agent vs. canonical divergences.

 Features 2, 3, and 5 share a single new abstraction (a TranscriptionCompanion
 class wrapping a whisper WebSocket). Feature 1 has its own seam (TwiML branching
 - after-hours config). Feature 4 is purely operator scripts — no runtime
 changes. So the natural ordering is foundation → companion users → distinct
 seam features → operator script.
